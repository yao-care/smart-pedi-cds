## 1. 備份範圍與策略（依零後端架構界定）

本系統無自有資料庫，故備份對象**不是**病患資料（病患／評估資料僅存於使用者瀏覽器 IndexedDB，由使用者裝置自行掌控，不在我方備份責任範圍）。我方需備份與可還原的資產為：

| 資產 | 真相源 / 備份位置 | 還原方式 | RPO | RTO 目標 |
|---|---|---|---|---|
| **原始碼** | Git repo（GitHub：`yao-care/smart-pedi-cds`）+ 維運者本機 clone | `git clone` / `git checkout <good-commit>` | 最後一次 push | ≤ 30 分鐘 |
| **線上網站** | GitHub Pages 由 `main` 自動建置部署 | 還原至已知良好 commit 後重新部署 | 同上 | ≤ 30 分鐘 |
| **網域 / DNS 設定** | 網域註冊商控制台 | 依紀錄重設 DNS | — | 依註冊商 |
| **建置設定 / 密鑰** | repo 內設定 + GitHub Actions secrets（如有） | 依文件重建 secrets | — | ≤ 1 小時 |

> 設計含義：因 repo 即真相源，「還原」本質上等同「以乾淨環境重建並重新部署」。最大風險不是資料遺失，而是**單一 GitHub 帳號失效或惡意變更**——故須維持至少一份**本機離線 clone** 作為獨立備份。

## 2. 還原測試程序（可重複執行的驗收腳本）

於乾淨環境執行下列步驟，全部通過即代表可還原：

1. 在全新目錄 `git clone` 該 repo（驗證遠端備份完整）。
2. 切換至指定的已知良好 commit。
3. `pnpm install`（驗證相依可重現安裝；lockfile 完整）。
4. `pnpm check` → 0 errors。
5. `pnpm test` → 全數通過。
6. `pnpm build` → exit 0 且 `postbuild` SEO 守門全過（產出 `dist/` 完整）。
7. 確認 `dist/` 可作為靜態站台正常開啟首頁與 `/assess`。

任一步失敗即代表還原能力缺口，須於下表記錄並修補。

## 4. 業務持續性備註

- 線上站台中斷時，因屬靜態內容，可暫時改由任一靜態主機（如其他 CDN / 物件儲存）以同一份 `dist/` 上線，DNS 切換後即恢復。
- 病患端離線能力：本系統具 Service Worker 與離線 sync queue，使用者端短暫離線不影響已載入之評估流程。
